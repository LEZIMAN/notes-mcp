## 🔧 工程地基

### 1. `requirements.txt` 与 `配置.md` 漂移
- **现象**:`配置.md §5` 列了 Web 后端依赖 `fastapi/uvicorn/httpx`,但 `requirements.txt` 实际没有。
- **根因**:文档与代码两个事实来源,改了一处忘了另一处。
- **修复**:补全 `requirements.txt` + 加 `scripts/dev.py check-sync`(白名单校验核心依赖/配置键齐全)+ 开发规范 §8「单一信息源」规则。
- **启示**:配置/依赖有多处真相时,必须有自动校验,不能靠自觉。

### 2. README 引用不存在的 `.env.example`
- **现象**:`README §快速开始` 写 `cp .env.example .env`,但文件不存在 → 照 README 跑不起来。
- **根因**:文档先行、文件后补,漏了落地。
- **修复**:照 `配置.md §4` 建 `.env.example`。

### 3. pip 直连 PyPI 网络中断(`IncompleteRead`)
- **现象**:`pip install` 下载 ruff 时连接中断。
- **根因**:国内访问 PyPI 不稳。
- **修复**:换清华镜像 `-i https://pypi.tuna.tsinghua.edu.cn/simple --retries 5`。

### 4. `git push` 连接重置(`Connection was reset`)
- **现象**:push 偶发失败。
- **根因**:GitHub 连接抖动。
- **修复**:本地 commit 已安全,稍后重试 `git push` 即可(不要重 commit)。

## 🛠 scripts/dev.py

### 5. `test` 子命令 `-q` 透传失败(argparse 局限)
- **现象**:`dev.py test -q` 报 `unrecognized arguments: -q`。试过 `nargs="*"` 和 `nargs=REMAINDER` 都不行。
- **根因**:argparse 把 `-` 开头的参数当 option 解析,找不到 `-q` 选项就报错;`REMAINDER` 对子命令首个 `-` 参数仍不可靠。
- **修复**:`main()` 里对 `test` 子命令**特判**——`if argv[0]=="test": return cmd_test(argv[1:])`,绕开 argparse,原样透传给 pytest。
- **启示**:argparse 的"透传未知参数"是已知坑,对 wrapper 脚本特判最稳。

### 6. E501 行长超 100(`add_parser` 链式)
- **现象**:逐行 `sub.add_parser(...).set_defaults(...)` 多行超 100 字符。
- **修复**:用注册表 + 循环批量注册,既短又 DRY。

## ✂️ chunker

### 7. 滑动窗口切块的边界(测试预期错)
- **现象**:`_split_text("0123456789", size=5, overlap=2)` 预期 3 块,实际 4 块。
- **根因**:step=3 时 `[0:5][3:8][6:10]` 后,`pos=9 < len=10` 还会切 `[9:10]="9"` 一块。**这是滑动窗口的正确行为**,不是 bug——是测试预期写错了。
- **修复**:改测试预期为 4 块。
- **启示**:先确认是"代码错"还是"理解错",别急着改实现。

## 📦 indexer

### 8. Path/str key 混用导致增量建库失效(致命 bug)⚠️
- **现象**:首次建库正常,但第二次 build 把所有文件当"新增+删除"(增量完全失效)。
- **根因**:`_scan` 返回 `dict[Path, float]`,`_load_db` 返回 `dict[str, FileState]`。`_diff` 里 `set(disk) - set(db)` 是 `set[Path] - set[str]`——**Path 和 str 永不相等**,集合运算认为完全不重叠 → 每次全量新增 + 全量删除。
- **修复**:内部 key **统一用 `str(path.resolve())`**(SQLite 天然存 str),`_scan`/`_load_db`/`_diff`/`_sync_sqlite` 全程 str,需要 Path 处再 `Path(path)`。
- **启示**:跨边界的 key 类型必须统一。mypy 能抓一部分(返回类型注解不符),但运行时 set 运算的隐式类型不匹配 mypy 不一定报。

### 9. bm25s 0.3.9 API:`save` 从模块函数变实例方法
- **现象**:`bm25s.save(retriever, dir)` 报 `module 'bm25s' has no attribute 'save'`。
- **根因**:bm25s 0.3+ 把 `save` 从模块级函数改成了 `BM25` 实例方法(`retriever.save(dir)`),`load` 仍是类方法。
- **修复**:`self._bm25.save(str(dir))`;装新依赖后先 `dir(bm25s)` / `dir(BM25)` 探测实际 API,别凭记忆。

### 10. Chroma `EphemeralClient` 同进程共享 + 固定 collection 名累积
- **现象**:`test_stats_resource` 断言 `total_files==3`,实际 `18`。
- **根因**:`chromadb.EphemeralClient()` 在**同进程内共享内存**,`get_or_create_collection("固定名")` 跨测试复用同一 collection,每次 fixture build 3 篇 → 累积(6 次测试 = 18 distinct sources,因 tmp_path 不同)。
- **修复**:测试 fixture 的 collection 名用 `tmp_path.name` 后缀(`f"server_{tmp_path.name}"`),保证每次唯一隔离。
- **启示**:EphemeralClient 不是"每次新内存",而是进程级共享。测试隔离要靠唯一命名,不能靠"新建 client"。

## 🌐 server

### 11. Searcher 缺 `collection` property
- **现象**:server 用 `searcher.collection` 报 `'Searcher' has no attribute 'collection'`。
- **根因**:Searcher 内部存 `self._collection` 但没暴露 property;Indexer 有 `collection` property,Searcher 漏了。
- **修复**:Searcher 加 `@property collection`。

### 12. fastmcp `read_resource` 返回 `list`(非对象)
- **现象**:`_resource_text` 用 `getattr(result, "contents")` 取不到,`json.loads` 失败。
- **根因**:fastmcp 3.x 的 `Client.read_resource` 直接返回 `list[TextResourceContents]`,不是带 `.contents` 的对象。
- **修复**:先判 `isinstance(result, list)`;第三方库返回结构别假设,先 `type(result)` / `dir(result)` 探测。

## ⚡ 性能

### 13. 建库慢(~5 分钟 / 1052 chunks)
- **现象**:首次 `index` 建 1052 chunk 耗时 ~5 分钟。
- **根因**:`OllamaEmbedder.embed` 是**单条**,indexer 循环 1052 次,每次一次 HTTP 往返(0.3s)。往返开销 × 1052 ≈ 5 分钟。
- **待优化**:加 `embed_batch(texts)` 批量嵌入(ollama `/v1/embeddings` 支持一次多 input),indexer 单文件 chunk 一次 embed。预期 1052/64 ≈ 17 次请求 → ~30 秒。
- **启示**:逐条 HTTP 是性能杀手;批量 + 摊薄往返开销是首选优化。

## 🧪 测试

### 14. 测试被项目根 `.env` 污染(`test_main_config_error_returns_2`)
- **现象**:验收时 `cp .env.example .env` 后,该测试失败(`assert 0 == 2`)。
- **根因**:测试 `monkeypatch.delenv("NOTES_DIR")` 只删环境变量,但 `Config.from_env` 的 `_find_env` 找到项目根 `.env` 并 `load_dotenv` 把 NOTES_DIR 注入回来 → notes_dirs 非空 → validate 通过 → 返回 0 而非 2。
- **修复**:测试 `monkeypatch.setattr("notes_mcp.config._find_env", lambda: None)`,不读项目根 .env。
- **启示**:测配置逻辑时要隔离项目根 `.env`——`load_dotenv` 会注入环境变量,绕过 `delenv`。

## 🌐 Web 后端(Phase 3a · Spring Boot)

### 16. Spring AI 连 fastmcp:相对路径 + 子进程 cwd 致 initialize 超时(多层排查)
- **现象**:Spring Boot(Spring AI MCP client stdio)启动 `Client failed to initialize`,20s 超时。手动测 fastmcp 响应 initialize **完全正常**。
- **根因**(层层剥):
  1. mcp-core(Java)spawn notes-mcp 子进程的 **`cwd` 配置字段没生效**(子进程实际 cwd = Java 进程 cwd,非项目根)。
  2. notes-mcp 的 `CHROMA_PATH=./data/chroma` 是**相对路径**,cwd 一变 → data 落到别处 → 触发**全量重建库**(embed 1052 chunk ~60s)。
  3. 子进程建库未完成 → mcp-core initialize 等不到响应 → 20s 超时。早期还叠了「`No module named notes_mcp`」(cwd 错导致 import 失败)和「streamable-http 跨实现互操作」两个干扰项。
- **修复**(组合拳):
  - `pip install -e .` 把 notes_mcp 装进 venv → 子进程任意 cwd 都能 import。
  - Config 加 `_resolve_path`:相对路径基于项目根解析 → data 永远指向已建库,不依赖 cwd。
  - Spring AI spawn args 加 `-u`(Python stdout 无缓冲,防父进程读不到)。
- **启示**:跨语言/跨进程 spawn 子进程时三件事——① **不依赖 cwd**(路径绝对化或包安装进环境);② **stdout 无缓冲**(否则父进程读不到响应);③ 先抓子进程 **STDERR** 确认它真的启动了(别只看父进程的笼统异常)。

## 🌐 Web 前端(Phase 3b · React)

### 17. CORS 只放行 GET/POST/OPTIONS,PIT 请求被 403
- **现象**:前端设置页保存笔记目录(PUT /api/settings)报 403 Forbidden。
- **根因**:`CorsConfig` 的 `.allowedMethods()` 只配了 `GET, POST, OPTIONS`,漏了 PUT 和 DELETE。
- **修复**:补全为 `GET, POST, PUT, DELETE, OPTIONS`。
- **启示**:加新 HTTP 方法时,必须同步检查 CORS 配置。

## 🌐 多 Provider(Phase 3b · Settings)

### 18. OpenAiChatOptions.apiKey() 不覆盖自动配置的密钥
- **现象**:切换到 DeepSeek Provider 后对话报 `401: Incorrect API key provided: unused`。明明 settings.json 里配了正确的 API key,但实际请求用的是 application.yml 里占位的 `unused`。
- **根因**:Spring AI 2.0.0 自动配置的 `OpenAiChatModel` 内部已有 `OpenAIClient`(用 `unused` 创建)。`ChatClient.prompt().options(OpenAiChatOptions.builder().apiKey(...))` 的 options 只是请求级参数,**不会覆盖底层 HTTP client 的认证凭据**。
- **修复**:不用自动配置的 `OpenAiChatModel`,改用 `OpenAiChatModel.builder().options(opts).build()` 动态创建,让 Spring AI 从 options 里读取 apiKey/baseUrl 构建新的内部 client。ProviderRouter 加缓存,同配置复用。
- **启示**:Spring AI 的 ChatOptions.apiKey() 设计上是用于「创建新 Model 时指定凭据」,不是「运行时覆盖已有 Model 的凭据」。多租户/多 Provider 场景需动态创建 ChatModel,不能依赖自动配置的单例。

### 19. 自定义 Provider 模型列表为空
- **现象**:切换到 DeepSeek 自定义 Provider 后,模型列表返回 `[]`,用户无法选择模型。
- **根因**:部分第三方 API(如 DeepSeek)不提供 `/v1/models` 端点,或端点路径不同。Spring AI 通过 OpenAI Java SDK 调 `/models` 拿不到数据。
- **修复**:前端模型管理 Tab 增加**手动输入模型名**功能——API 返回空列表时,显示文本框让用户直接输入模型 ID(如 `deepseek-chat`),回车或点击即可使用。
