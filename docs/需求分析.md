# notes-mcp 需求分析文档

> 版本 v0.1 · 2026-07-17 · 项目实战 P3
> 配套:[README](../README.md) · [设计文档](./设计文档.md)

## 1. 背景

### 1.1 问题
markdown 笔记越积越多,而 AI 工具(Cursor / Claude / Obsidian)彼此隔离,各自不知道用户的笔记内容。每让一个工具「懂我的笔记」就要重做一遍 RAG(切块 → 嵌入 → 检索),碎片化、重复劳动。

### 1.2 机会
**MCP**(Model Context Protocol,Anthropic 2024-11 开放协议)提供了「工具标准化供给」——把能力封成 MCP server,任意 MCP client 接上即可复用。**一处实现,处处可用**。

### 1.3 定位
`notes-mcp` = 把「一个或多个 markdown 笔记目录」变成**标准 MCP 知识助手 server** 的工具。**不只「查得到」笔记,更把笔记变成可用结果**(讲解 / 复习提纲 / 自测题 / 对比表)——分三层:**找(search)→ 拿(get)→ 用(prompts)**。既是个人学习笔记⑳(MCP 协议精读)的工程落地,也是可开源的通用工具。

> 💡 核心洞察:**检索 ≠ 有用**。笔记的价值是「学/用/解决问题」,不是「被查」。所以本项目不止做检索(找),更做应用层(用)——用 Prompts 把检索到的笔记组织成可用的产物。

## 2. 目标 / 非目标

### 2.1 目标(in scope)

| ID | 目标 |
|---|---|
| G1 | **通用**:任意 markdown 目录(支持多个)→ MCP server,配置即用 |
| G2 | **三大原语齐全**:Tools / Resources / Prompts 都有 |
| G3 | **检索质量**:hybrid(语义 + 关键词 + RRF 融合)+ 溯源 |
| G3.5 | **不止检索,产出可用结果(应用层)**:用 Prompts 把笔记变成讲解 / 复习 / 自测 / 对比(找→拿→用闭环)——「记笔记有什么用」的答案 |
| G4 | **可复用**:被 Claude Desktop / Cursor / inspector / 自写 agent 调 |
| G5 | **工程规范**:自动增量建库、配置化、双传输、详细注释 |
| G6 | **教学**:与笔记⑳ / 07 / 08 / 09–11 呼应 |

### 2.2 非目标(out of scope)

| ID | 不做 | 理由 |
|---|---|---|
| N1 | 云 SaaS / 多租户 / 用户管理 | 单机个人工具,控制体量 |
| N2 | 向量库托管 | 用本地 Chroma,零运维 |
| N3 | 原生桌面/移动 App | 只做 **Web 图形界面**(FastAPI + React),不做 Electron/移动端 |
| N4 | rerank 大模型 | 用轻量 RRF;留扩展接口 |

## 3. 用户与场景

### 3.1 用户画像

- **P1 自己**:AI Agent 方向学习者,markdown 笔记多,日常用多个 AI 工具。
- **P2 开源使用者**:其他想把自己的笔记/Obsidian 库 MCP 化的人。

### 3.2 核心用例

| UC | 角色 | 目标 |
|---|---|---|
| UC1 | P1 | 在 Claude Desktop 直接问「RAG 和 ReAct 的联系」,server 查笔记返回带出处 |
| UC2 | P1 | Cursor 写代码时查学习笔记里的概念 |
| UC3 | P1 | 自写 agent 多跳:查笔记 + 读文件 + 总结 |
| UC4 | P2 | inspector 调试检索质量、对比 hybrid vs 纯语义 |

### 3.3 典型多跳任务
> 「先查笔记里 RAG 是什么,再读 notes-mcp 的 requirements.txt,总结」
> → agent 调 `search_notes`(自己的 server)+ `read_file`(filesystem server)

## 5. 非功能需求(NFR)

| 维度 | 要求 |
|---|---|
| 性能 | 建库百篇笔记秒级;单次检索 < 1s(本地,待实测) |
| 可维护 | 模块化、配置外置、**教学型详细中文注释** |
| 可移植 | Windows 11 优先;用 `pathlib`,跨平台 |
| 安全 | 路径遍历防护;stdio 下不 `print`;密钥走 `.env` 不硬编码 |
| 可开源 | 清晰 README + `.env.example` + 友好 license |
| 依赖 | 尽量轻量(核心:`fastmcp` / `chromadb` / `bm25s` / `jieba`) |

## 6. 约束与假设

- 本地 **ollama** 已部署(`qwen3:8b` 生成 + `bge-m3` embedding)
- Python 3.10+
- 笔记为 markdown(`.md`)
- 单机 / 个人使用(非多用户)
- `node` / `npx` 可用(消费侧接官方 filesystem/fetch server)

---

## 7. 验收标准

- **server**(inspector):list tools 看到 `search_notes` / `get_note`(+`list_topics`);list resources 看到 `notes://stats` 和模板 `notes://note/{title}`;调 `search_notes("RAG 的四步流水线")` 返回带 source 出处的 hybrid 结果。
- **agent**:完成 3 个多跳 demo(查笔记 / 读文件 / 混合)。
- **[P1] HTTP**:`curl http://localhost:8765/health` 通;client 连 streamable-http 成功。
- **[P1] tests**:RRF 融合单测通过。

---

## 8. 风险(摘自设计阶段)

- 绝对路径(MultiServerMCPClient args / claude config)
- async 生命周期(client 存活、`get_tools` 后不退上下文)
- stdio 下 `print` 污染协议 → 用 logging
- RRF 两路 id 对齐(BM25 必须 chunk 级)
- 中文 BM25 停用词(待测试)
- resource 模板路径遍历(`is_relative_to` 校验)
- `bm25s[core]` 在 Windows 编译(备选 `rank_bm25`)
- ollama IPv6(用 `127.0.0.1`)
